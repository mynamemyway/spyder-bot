# feat(UX): Расширение шаблона ввода размера и исправление логики FSM

## Краткое описание

Данное обновление направлено на улучшение пользовательского опыта (UX) и исправление критического бага в логике навигации по воронке продаж. Были добавлены новые, более интуитивные разделители для ввода размеров, а также скорректирована работа с конечным автоматом (FSM) для обеспечения консистентного поведения бота при возврате на предыдущие шаги.

## Основные изменения

### 1. Расширение разделителей для ввода размера

Для повышения гибкости ввода размеров баннера были добавлены новые разделители. Теперь, помимо `x` и `х`, пользователь может использовать символ `*` и слово `на` (в любом регистре).

-   **Файл:** `app/handlers/user_handlers.py`
-   **Логика:** Регулярное выражение в декораторе `@router.message` и в теле обработчика `handle_size_input` было модифицировано.
-   **Реализация:** Центральная часть выражения `[xх]` была заменена на не-захватывающую группу `(?:[xх]|\*|на)`. Это позволяет парсеру корректно распознавать следующие форматы:
    -   `3 * 2`
    -   `3 на 2`
    -   `3 НА 2`
    -   `3,5 На 2.1`

### 2. Исправление логики сброса состояния FSM при навигации назад

Была обнаружена и исправлена ошибка, при которой состояние FSM не сбрасывалось корректно при возврате на общие шаги меню (выбор категории, выбор материала).

-   **Проблема:**
    1.  При нажатии кнопки "Назад" для перехода на шаги, не требующие специфического текстового ввода, использовался вызов `await state.clear()`.
    2.  Это полностью очищало все данные пользователя в FSM, включая `calculator_message_id`, что приводило к ошибкам `Calculator message ID not found` при попытке обновить виджет.
    3.  Даже если бы ошибки не было, сохранение состояния `OrderState.entering_size` приводило к тому, что обработчик `handle_size_input` перехватывал текстовые сообщения, которые должны были уходить в RAG-цепочку.
-   **Решение:**
    -   **Файл:** `app/handlers/user_handlers.py`
    -   **Логика:** В функции `navigate_catalog` для всех уровней, представляющих собой общие меню (`level=0`, `level=1`, `level=2`), вызов `await state.clear()` был заменен на `await state.set_state(None)`.
    -   **Объяснение:** В отличие от `clear()`, метод `set_state(None)` сбрасывает только *состояние* конечного автомата, но **сохраняет все данные** (`data`), включая `calculator_message_id`.

## Результат

1.  **Улучшенный UX:** Пользователи могут вводить размеры более естественным способом.
2.  **Стабильность воронки:** Устранены ошибки, связанные с потерей ID виджета.
3.  **Консистентное поведение:** Теперь при возврате на общие шаги меню бот корректно обрабатывает текстовые сообщения как запросы к AI, а не как попытку ввода размера, что соответствует ожиданиям пользователя.
