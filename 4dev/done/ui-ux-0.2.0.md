# План рефакторинга: "Живой калькулятор" и гибкая воронка

**Цель:** Реализовать концепцию "живого калькулятора" — единого, постоянно обновляемого сообщения с расчетом, которое не прерывает основной диалог с AI. Расширить воронку продаж, добавив новые шаги, и позволить пользователю задавать вопросы консультанту на любом этапе воронки.

---

### **Этап 1: Модификация Состояний и Клавиатур**

На этом этапе мы расширим конечный автомат (FSM) и создадим новые клавиатуры для дополнительных шагов воронки.

#### **1.1. Обновление FSM States**
*   **Цель:** Добавить новые состояния для всех шагов воронки, включая количество и финальное оформление.
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:** В класс `OrderState` добавить состояния: `choosing_print_quality`, `choosing_ink_type`, `entering_quantity`, `choosing_urgency`, `choosing_delivery`, `choosing_processing_type`, `entering_delivery_address`, `entering_comment`.

#### **1.2. Создание новых клавиатур**
*   **Цель:** Создать клавиатуры для новых шагов воронки.
*   **Файл:** `app/keyboards.py`
*   **Логика:**
    1.  Создать функцию `get_print_quality_keyboard()`, которая будет генерировать кнопки для выбора DPI (**540, 720, 1440**).
    2.  Создать функцию `get_ink_type_keyboard()` (если решим вернуть).
    3.  Создать функцию `get_urgency_keyboard()` с кнопками "Обычный (2-3 дня)" и "Срочный (1 день)".
    4.  Создать функцию `get_delivery_keyboard()` с кнопками "Да" и "Нет".
    5.  Создать функцию `get_processing_type_keyboard()` с кнопками для выбора типа обработки (люверсы, карманы и т.д.).
    6.  Все клавиатуры воронки будут содержать кнопку "Назад" для навигации.

#### **1.3. Синхронизация клавиатуры и идентификаторов материалов (Новый пункт)**
*   **Цель:** Привести клавиатуру выбора материалов и `item_id` в соответствие с актуальным прайс-листом `calculator_from_website_banners.md`.
*   **Файл 1:** `app/keyboards.py`
    *   **Логика:** В функции `get_banner_materials_keyboard` обновить тексты кнопок и `item_id` в `CatalogCallback` на новые семантические идентификаторы (`frontlit_440`, `frontlit_cast_530`, `blackout`, `backlit`, `mesh`).
*   **Файл 2:** `app/handlers/user_handlers.py`
    *   **Логика:** В обработчиках `navigate_catalog` и `handle_size_input` обновить словари `material_map` для корректного сопоставления новых `item_id` с их русскоязычными названиями.

### **Этап 2: Рефакторинг Обработчиков и Логики Воронки**

Это основной этап, на котором мы реализуем новую логику "живого калькулятора" и гибкого диалога.

#### **2.1. Реализация "Живого калькулятора"**
*   **Цель:** Отделить сообщение-калькулятор от основного диалога и сделать его единым обновляемым виджетом.
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:**
    1.  **В `handle_catalog`:** Вместо простого `message.answer_photo`, мы будем сохранять результат вызова в переменную (`calculator_message = await message.answer_photo(...)`) и немедленно сохранять его ID в FSM-состоянии: `await state.update_data(calculator_message_id=calculator_message.message_id)`.
    2.  **В `navigate_catalog`:** Перед вызовом `_edit_message` или `_edit_caption`, обработчик будет извлекать `calculator_message_id` из состояния.
    3.  **В `_edit_message` и `_edit_caption`:** Сигнатуры этих функций будут изменены, чтобы принимать не `message: Message`, а `chat_id: int`, `message_id: int` и `bot: Bot`. Внутри они будут использовать `bot.edit_message_text`, `bot.edit_message_caption` и т.д., обращаясь к сообщению по его ID. Это отвяжет логику редактирования от конкретного `callback_query.message`.

#### **2.2. Реализация гибкого диалога в воронке**
*   **Цель:** Позволить пользователю задавать вопросы AI, находясь в состоянии ожидания ввода данных (например, размера).
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:**
    1.  **Модификация `handle_size_input`:** Этот обработчик больше не будет ловить все текстовые сообщения в состоянии `OrderState.entering_size`. Вместо `F.text` мы будем использовать `F.text.regexp(r"^\s*(\d+(?:[.,]\d+)?)\s*[xх]\s*(\d+(?:[.,]\d+)?)\s*$")`, чтобы он срабатывал только на формат "ширина x высота".
    2.  **Создание нового обработчика:** Создать новый обработчик `handle_question_in_funnel(message: Message, bot: Bot, state: FSMContext)`, который будет срабатывать на `F.text` во всех состояниях воронки (`OrderState.entering_size`, `OrderState.choosing_print_quality` и т.д.). Этот обработчик будет вызывать `process_query`, передавая в него вопрос пользователя. Таким образом, любой текст, не соответствующий формату ожидаемых данных, будет обработан как обычный вопрос к AI.

#### **2.3. Расширение воронки продаж**
*   **Цель:** Интегрировать новые шаги выбора качества печати и типа чернил.
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:**
    1.  **В `handle_size_input`:** После успешного ввода размера, переводить пользователя в состояние `OrderState.choosing_print_quality` и обновлять калькулятор.
    2.  **В `navigate_catalog`:** Добавить обработку колбэков для новых уровней воронки.
        *   **Логика "Живого калькулятора":** При движении вперед по шагам, влияющим на цену (качество -> срочность -> количество), сохранять выбор в FSM, переводить в следующее состояние и обновлять калькулятор.
        *   При движении назад (нажатие кнопки "Назад"): **явно устанавливать FSM-состояние, соответствующее предыдущему шагу воронки.** Например, если пользователь был на шаге выбора чернил и нажал "Назад" к выбору качества печати, FSM-состояние должно быть установлено в `OrderState.choosing_print_quality`. Если пользователь возвращается к выбору материала, FSM-состояние должно быть установлено в `OrderState.entering_size`. При возврате к началу каталога или главному меню, FSM-состояние должно быть очищено (`state.clear()`).
    3.  **Переход к оформлению:** После выбора всех ценообразующих параметров, в калькуляторе появится кнопка "Оформить заказ".
    4.  **Логика оформления заказа:** При нажатии на "Оформить заказ", бот последовательно запрашивает:
        *   Тип обработки (клавиатура `get_processing_type_keyboard`).
        *   Необходимость доставки (клавиатура `get_delivery_keyboard`).
        *   Если доставка нужна, переходит в состояние `entering_delivery_address` и просит ввести адрес.
        *   Просит ввести комментарий к заказу (состояние `entering_comment`).
    5.  На финальном шаге производить итоговый расчет, выводить полный состав заказа и очищать состояние (`state.clear()`).

#### **2.4. Реализация нового калькулятора цен (Новый пункт)**
*   **Цель:** Создать модуль расчета цен, учитывающий все параметры из нового прайса.
*   **Файл:** `app/core/pricing.py`
*   **Логика:**
    1.  Создать новую структуру данных (словарь), отражающую прайс из `4dev/info/calculator_from_website_banners.md`. Ключами будут `(material, dpi)`.
    2.  Модифицировать функцию `calculate_price`, чтобы она принимала все необходимые параметры: `material`, `width`, `height`, `dpi`, `quantity`, `is_urgent`, `needs_delivery`.
    3.  Внутри функции реализовать логику:
        *   Найти базовую цену по `material` и `dpi`.
        *   Рассчитать стоимость за площадь (`price * width * height`).
        *   Умножить на количество (`* quantity`).
        *   Применить наценку +50% за срочность.
        *   Добавить стоимость доставки.

# Refactor: "Live Calculator" and Flexible Sales Funnel (v0.2.0)

## Краткое описание

Этот pull request знаменует собой завершение второго крупного этапа рефакторинга, направленного на кардинальное улучшение пользовательского опыта и расширение функциональности воронки продаж. Ключевые нововведения — реализация концепции "живого калькулятора" и создание гибкой, многошаговой воронки с возможностью консультации с AI на любом этапе.

## I. Реализация "Живого калькулятора"

**Проблема:** Ранее каждый шаг воронки создавал новое сообщение, что засоряло чат и мешало пользователю видеть актуальный расчет.

**Решение:** Внедрена концепция "живого калькулятора" — единого, постоянно обновляемого сообщения-виджета.

-   **Централизованное управление виджетом:**
    -   При старте воронки (`/catalog`) ID сообщения-калькулятора сохраняется в FSM-состоянии (`calculator_message_id`).
    -   Создана интеллектуальная функция-обертка `_update_calculator_widget`, которая инкапсулирует всю логику обновления UI. Она автоматически определяет, нужно ли менять изображение (используя "тяжелую" операцию `_edit_message`) или достаточно обновить только текст (используя "легкую" `_update_calculator_message`).
    -   Все обработчики воронки теперь вызывают `_update_calculator_widget`, передавая желаемое состояние, что делает код чистым и декларативным.
-   **Управление жизненным циклом виджета:**
    -   Реализовано автоматическое удаление старого виджета при создании нового (команды `/start`, `/catalog`), что предотвращает накопление неактуальных сообщений.
-   **Визуальное сопровождение:**
    -   Добавлены и интегрированы кастомизированные изображения для ключевых этапов воронки: выбор продукта, расчет и подтверждение заказа (`BANNERS_PHOTO_PATH`, `CALCULATION_PHOTO_PATH`, `ORDER_PHOTO_PATH`).
    -   Исправлен баг, при котором изображения "залипали" при навигации назад. Теперь `_update_calculator_widget` корректно отслеживает и меняет изображение на каждом шаге.

## II. Расширение и гибкость воронки продаж (FSM)

**Проблема:** Воронка была короткой и негибкой, не позволяя пользователю отвлекаться на уточняющие вопросы.

**Решение:** Воронка была значительно расширена и сделана интерактивной.

-   **Новая последовательность шагов:** Реализована полная и логичная последовательность выбора параметров для заказа баннера:
    1.  Выбор материала.
    2.  Ввод размера.
    3.  Выбор качества печати (DPI).
    4.  Выбор типа постобработки.
    5.  Выбор количества (включая ручной ввод).
    6.  Выбор срочности заказа.
    7.  Выбор способа получения (доставка/самовывоз).
    8.  Финальный расчет и подтверждение заказа.
-   **Гибкий диалог:**
    -   Реализован обработчик `handle_question_in_funnel`, который позволяет пользователю на любом этапе воронки задать вопрос AI в свободной форме.
    -   Обработчики ввода данных (например, размера) теперь срабатывают только на конкретный формат (`regexp`), передавая все остальные текстовые сообщения на обработку AI-консультанту. Это позволяет пользователю уточнять детали, не выходя из процесса заказа.
-   **Обновленный калькулятор цен:**
    -   Модуль `app/core/pricing.py` был полностью переработан.
    -   Создана новая структура прайс-листа на основе `calculator_from_website_banners.md`.
    -   Функция `calculate_price` теперь учитывает все новые параметры: `dpi`, `quantity`, `is_urgent`, `needs_delivery`.

## III. Уведомление менеджера и завершение заказа

**Проблема:** Процесс заказа обрывался на ссылке на менеджера без фиксации самого заказа.

-   **Обратная связь для пользователя:** Сообщение-калькулятор обновляется в последний раз, показывая статус "Заказ принят!" и предоставляя кнопку для прямой связи с менеджером.

В результате этих изменений бот получил мощную, гибкую и интуитивно понятную воронку продаж, которая значительно улучшает пользовательский опыт и автоматизирует процесс сбора информации для заказа.
