# Детальный План Рефакторинга: AI-Консультант "Пиранья"

**Цель:** Пошаговая трансформация Telegram-бота "Portfolio AI" в AI-консультанта по продажам для типографии "Пиранья". План сфокусирован на сохранении стабильного ядра RAG и последовательной адаптации компонентов под новую бизнес-логику, начиная с MVP для продукта "Баннеры".

---

## Этап 1: Адаптация Архитектуры и Конфигурации

**Задача:** Подготовить проект к работе с новыми данными и интегрировать процесс индексации в сборку Docker-образа для создания самодостаточного и переносимого приложения.

#### **1.1. Наполнение Базы Знаний (MVP)**
*   **Цель:** Создать минимально необходимую базу знаний для консультаций по продукту "Баннеры".
*   **Действие:** Создание нового файла.
*   **Файл:** `app/knowledge_base/banners.md`
*   **Логика:** Наполнить файл структурированной информацией о баннерах: доступные материалы (глянцевый, матовый), размеры, описание, преимущества, ответы на частые вопросы. Информацию взять с сайта `mir-pr.ru`.

#### **1.2. Интеграция Индексации в Сборку Образа**
*   **Цель:** Автоматизировать создание векторного индекса `ChromaDB` при сборке Docker-образа. Это устраняет необходимость ручного запуска скрипта индексации и гарантирует, что образ всегда содержит актуальный индекс.
*   **Действие:** Модификация файла.
*   **Файл:** `Dockerfile`
*   **Логика:** Добавить команду `RUN python -m app.core.rag` перед финальной командой `CMD`. На этапе `docker-compose build` эта команда выполнит скрипт `rag.py`, который загрузит документы из `app/knowledge_base`, векторизует их и сохранит индекс в директорию `chroma_db` внутри образа.

#### **1.3. Обновление Документации**
*   **Цель:** Адаптировать `README.md` для отражения новой цели проекта и упрощенного процесса запуска.
*   **Действие:** Модификация файла.
*   **Файл:** `README.md`
*   **Логика:**
    1.  Изменить заголовок и описание проекта на "AI-консультант 'Пиранья'".
    2.  Обновить раздел инструкций по запуску: удалить шаг с ручным запуском индексации (`docker-compose run ... rag`) и указать, что теперь достаточно выполнить `docker-compose build` и `docker-compose up`.

**Задача:** Изменить "личность" и поведение AI, чтобы он соответствовал роли продавца-консультанта.

#### **2.1. Обновление Системного Промпта**
*   **Цель:** Задать AI новую роль, цели и стиль общения.
*   **Действие:** Модификация файла.
*   **Файл:** `.env.example`
*   **Логика:** Изменить значение переменной `SYSTEM_PROMPT`. Новый промпт должен инструктировать LLM о его роли (продавец-консультант типографии "Пиранья"), задачах (вести по воронке, консультировать, помогать с выбором) и стиле общения (клиентоориентированный, экспертный, дружелюбный).

## Этап 3: Рефакторинг Пользовательского Интерфейса (UI)

**Задача:** Заменить UI, ориентированный на портфолио, на UI, реализующий воронку продаж для выбора баннеров.

#### **3.1. Обновление Команд Меню**
*   **Цель:** Заменить команды в меню Telegram на релевантные для бота-консультанта.
*   **Действие:** Модификация файла.
*   **Файл:** `app/ui_commands.py`
*   **Логика:** Изменить список команд. Предлагаемый набор:
    *   `/start` - Начало диалога / Перезапуск бота.
    *   `/catalog` - Показать каталог продукции (начало воронки).
    *   `/reset` - Сбросить контекст диалога.
    *   `/help` - Помощь.

#### **3.2. Переработка Клавиатур (Keyboards)**
*   **Цель:** Создать новые inline-клавиатуры для навигации по воронке продаж.
*   **Действие:** Модификация файла.
*   **Файл:** `app/keyboards.py`
*   **Логика:**
    1.  Удалить старые клавиатуры, связанные с портфолио (`get_main_keyboard`, `get_projects_keyboard` и т.д.).
    2.  Создать новую фабрику колбэков, например, `CatalogCallback`.
    3.  Реализовать функции для генерации клавиатур на каждом шаге воронки:
        *   `get_start_keyboard()`: Клавиатура для стартового сообщения с кнопкой "Каталог".
        *   `get_catalog_keyboard()`: Клавиатура для выбора категории продукта (начнем с одной кнопки "Баннеры").
        *   `get_banner_materials_keyboard()`: Выбор материала для баннера на основе данных с сайта (Баннерная ткань, Баннерная сетка, Транслюцентный, Blackout).
        *   `get_banner_sizes_keyboard()`: Выбор размера.
        *   Клавиатура с кнопкой "Назад" для возврата на предыдущий шаг.

#### **3.3. Переработка Обработчиков (Handlers)**
*   **Цель:** Реализовать логику обработки новых команд и навигации по клавиатурам.
*   **Действие:** Модификация файла.
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:**
    1.  Полностью переработать файл, удалив старые обработчики (`handle_start`, `handle_main_menu_button` и т.д.), связанные с логикой портфолио.
    2.  Реализовать новый обработчик `handle_start`, который отправляет приветственное сообщение и `get_start_keyboard`.
    3.  Реализовать обработчик команды `handle_catalog`, который инициирует воронку продаж.
    4.  Создать единый обработчик для `CatalogCallback`, который будет маршрутизировать логику в зависимости от `callback_data`. Он будет управлять сменой сообщений и клавиатур, последовательно проводя пользователя по шагам воронки (выбор продукта -> материала -> размера).
    5.  Обработчик текстовых сообщений `handle_message` остается без изменений, так как он делегирует обработку вопросов ядру RAG.

## Этап 4: Реализация Калькулятора Цен (MVP)

**Задача:** Добавить базовую логику расчета стоимости заказа на основе выбора пользователя.

#### **4.1. Создание Модуля Расчета Цен**
*   **Цель:** Инкапсулировать логику прайс-листа и расчета стоимости в отдельном модуле.
*   **Действие:** Создание нового файла.
*   **Файл:** `app/core/pricing.py`
*   **Логика:**
    1.  Создать структуру данных для хранения цен (например, вложенный словарь Python), где ключами будут параметры заказа (продукт, материал, размер).
    2.  Реализовать функцию `calculate_price(product, material, size)`, которая принимает параметры заказа и возвращает итоговую стоимость из прайс-листа.

#### **4.2. Интеграция Калькулятора в Воронку**
*   **Цель:** Собирать выбор пользователя и на финальном шаге выводить рассчитанную цену.
*   **Действие:** Модификация файла.
*   **Файл:** `app/handlers/user_handlers.py`
*   **Логика:**
    1.  **Создание Состояний (States):** Определить группу состояний `OrderState` в `aiogram.fsm.state` для отслеживания этапов сбора информации. Для MVP будет создано состояние `entering_size`.
    2.  **Модификация обработчика колбэков (`navigate_catalog`):**
        *   При выборе материала (`level == 3`) сохранить `material_id` в хранилище FSM (`state.update_data`).
        *   Перевести пользователя в состояние `OrderState.entering_size` с помощью `state.set_state()`, активируя режим ожидания ввода размера.
    3.  **Создание обработчика для ввода размера:**
        *   Создать новый обработчик, срабатывающий только в состоянии `OrderState.entering_size`.
        *   Использовать регулярное выражение для парсинга размеров из сообщения.
        *   При успехе: извлечь `material_id` из FSM, вызвать `calculate_price`, отправить результат и очистить состояние (`state.clear()`).
        *   При неудаче: отправить сообщение об ошибке формата.
    4.  **Изоляция обработчиков:** Добавить `StateFilter(None)` к основному текстовому обработчику `handle_message`, чтобы он не перехватывал сообщения, предназначенные для FSM.
